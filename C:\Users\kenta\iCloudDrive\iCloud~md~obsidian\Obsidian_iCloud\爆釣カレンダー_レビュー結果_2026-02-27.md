### 2. 【超重大】スクレイピング時の数値パースバグ

**問題**: 釣具屋サイトの「範囲表記(例: 300-999匹)」を数値化する際、ハイフンを削除して結合してしまい、異常な大量釣果データになっている

#### 発見の経緯
公式サイトで確認したところ、以下のような変換バグが発覚:
- **サイト表記**: 300-999匹 (範囲表記)
- **DB格納値**: 300,999匹 (!!!)

#### 証拠データ
```sql
-- 1日の最大釣果トップ10(すべてパースバグの可能性)
2023/01/16 東陽町店 ワカサギ: 7,001,600匹 ← 「700-1600匹」
2016/02/07 高崎店 ワカサギ: 5,001,000匹 ← 「500-1000匹」
2013/02/19 能代店 ワカサギ: 4,501,111匹 ← 「450-1111匹」
2010/03/12 新富山空港通り店 ホタルイカ: 300,999匹 ← 「300-999匹」(確認済)
2014/12/28 池袋店 ヘラブナ: 999,999匹 ← 「999匹以上」または「999-999匹」
2011/01/23 甲府昭和店 ワカサギ: 999,999匹
2010/10/17 甲府昭和店 ワカサギ: 999,999匹

-- 不自然に頻出する数値パターン(範囲表記の証拠)
1020匹: 137回出現 ← 「10-20匹」
510匹: 178回出現 ← 「5-10匹」
1015匹: 84回 ← 「10-15匹」
2030匹: 82回 ← 「20-30匹」
```

**原因コード**: `src/scraper_johshuya.ts:119`

```typescript
count: parseInt(countStr.replace(/[^0-9]/g, '')) || 0,
//                              ↑
// ハイフン(-)も削除されるため、「300-999」→「300999」になる!!!
```

**影響**:
- **ワカサギ700万匹** など、物理的にあり得ない釣果がDBに保存されている
- 機械学習モデルが「釣果数の正常範囲」を完全に誤学習
- パーセンタイルスコア計算が破綻(score_distributionに極端な値が含まれる)
- 本来50点の日でも99点になってしまう根本原因

**魚種別総釣果(海カテゴリ)の異常**:
```sql
スルメイカ: 1,571,758匹
アジ: 1,266,636匹
ヤリイカ: 680,242匹
ハタハタ: 646,608匹
ホタルイカ: 632,169匹  ← 富山の特産(パースバグ多数含む)
```

**対策案**:
1. **スクレイピングコードの修正(最優先!!)**
   ```typescript
   // 範囲表記の場合は中央値または下限値を採用
   const countStr = $(tr).find('td').last().text().trim();
   let count = 0;

   if (countStr.includes('-')) {
       // 範囲表記: "300-999" → 下限値300を採用
       const [min, max] = countStr.split('-').map(s => parseInt(s.replace(/[^0-9]/g, '')));
       count = min || 0; // または (min + max) / 2 で中央値
   } else {
       count = parseInt(countStr.replace(/[^0-9]/g, '')) || 0;
   }
   ```

2. **既存データのクリーニング(DB修正)**
   - 1万匹以上のレコードを手動確認して修正または削除
   - 特にワカサギ、イカ類、ホタルイカなど群れで釣れる魚種を重点的にチェック

3. **学習データの前処理**
   - 釣果数の上限クリッピング(例: 1日100匹以上は異常値として除外)
   - 群れで釣れる魚種(ホタルイカ、スルメイカ、ハタハタ、ワカサギ等)を学習対象から除外
   - 対数変換による外れ値の影響軽減

### 3. 【重大】相関係数が極端に低い

**問題**: 全ての特徴量と釣果数の相関が0.02以下と、ほぼ無相関状態

#### 分析結果

**正常な相関の目安**: 0.3以上あれば予測に有効、0.1未満は「ほぼ関係ない」レベル

**原因推測**:
1. 上記の「地理的汚染」「外れ値混入」により、本来の相関が隠されている
2. 釣果データの記録バイアス(釣れた日だけ報告、釣具屋が盛る等)
3. 「釣果数」ではなく「釣果率(CPUE)」を使うべき可能性

**影響**:
- AIモデルが「適当な予測」しかできない
- 結果的にほぼ全日99点という異常な出力になる

## 💡 根本的な設計の見直し提案

### 目的変数の再検討

**現状**: 釣果数(catch_count)を予測
**問題**: 釣り人の数や記録バイアスの影響を強く受ける

**提案**: CPUE(Catch Per Unit Effort)を使用
- CPUE = 釣果数 / 釣り人数(または釣り時間)
- 施設データには`visitors`(入場者数)があるため計算可能
- より「釣りやすさ」を反映した指標になる

---

### モデルアーキテクチャの再検討

**現状**: 2段階AI(海況予測 → 釣果予測)
**懸念**: 海況予測の誤差が釣果予測に累積する

**代替案**: End-to-Endモデル
- 気象データ → 釣果 を直接予測
- 海況データは「説明変数の一つ」として補助的に使用
- モデルがシンプルになり、デバッグしやすい
